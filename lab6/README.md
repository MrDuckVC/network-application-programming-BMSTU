## 3. Теория и реализация (Примеры из кода)

### 3.1. Сравнение Callback Hell и Async/Await

**Теория:** Использование `XMLHttpRequest` заставляло разработчиков передавать функции обратного вызова (колбэки) аргументами в сетевой слой. При построении сложных цепочек зависимых запросов это приводило к синтаксическому тупику **"Callback Hell"** (или пирамиде погибели), ухудшая читаемость и усложняя перехват ошибок.

Введение объектов `Promise` и синтаксического сахара `async/await` позволило писать асинхронный код в линейном, синхронном стиле. Конструкция `await` приостанавливает выполнение функции в контрольной точке до тех пор, пока промис не перейдет из состояния `Pending` в `Fulfilled` (возвращая результат вычисления) или в `Rejected` (генерируя исключение, перехватываемое блоком `catch`).

### 3.2. Реализация обертки над Fetch API (Модуль Ajax)

**Теория:** Функция `fetch()` предоставляет современный интерфейс для работы с HTTP-запросами. Важной особенностью `fetch` является то, что возвращаемый им промис **не отклоняется (not rejected)** при получении серверных статус-кодов ошибок (таких как `404 Not Found` или `500 Internal Server Error`). Промис уходит в состояние `Rejected` только при физическом сбое сети или невозможности достучаться до сервера. В связи с этим в обертку внедрена ручная верификация свойства `response.ok`.

**Код обновленного модуля `lab6/client/src/modules/ajax.js`:**

```javascript
class Ajax {
    // Асинхронный GET запрос
    async get(url) {
        try {
            const response = await fetch(url);
            // fetch не генерирует исключение при статусах 4xx/5xx, проверяем флаг ok (статусы 200-299)
            if (!response.ok) {
                throw new Error(`Ошибка сервера: ${response.status}`);
            }
            return await response.json(); // Десериализация тела ответа
        } catch (error) {
            console.error("GET error в сетевом слое:", error);
            throw error; // Пробрасываем ошибку выше в UI-компонент
        }
    }

    // Асинхронный PATCH запрос
    async patch(url, data) {
        try {
            const response = await fetch(url, {
                method: 'PATCH',
                headers: {
                    'Content-Type': 'application/json' // Передача метаданных о типе контента
                },
                body: JSON.stringify(data) // Сериализация JS-объекта в JSON-строку
            });
            if (!response.ok) {
                throw new Error(`Ошибка PATCH: ${response.status}`);
            }
            return await response.json();
        } catch (error) {
            console.error("PATCH error в сетевом слое:", error);
            throw error;
        }
    }
}
export const ajax = new Ajax();

```

### 3.3. Асинхронные компоненты страниц (Использование await)

Благодаря интеграции `async/await`, логика обработки ответов в UI-компонентах избавилась от вложенных функций. Ошибки отсутствия связи с бэкендом теперь обрабатываются централизованно через конструкции `try...catch`.

**Пример обработки сетевого состояния в `lab6/client/src/pages/main/index.js`:**

```javascript
async getData() {
    const innerContainer = document.getElementById('carousel-inner-container');

    try {
        // Линейное ожидание данных без использования колбэков
        const data = await ajax.get(starshipUrls.getStarships());

        this.currentData = data;
        this.renderCarouselItems(); // Отрисовка UI при успехе
    } catch (error) {
        console.error("Компонент MainPage зафиксировал сбой:", error);

        // Отрисовка понятного деградационного интерфейса (Fall-back UI) для пользователя
        innerContainer.innerHTML = `
            <div class="d-flex flex-column justify-content-center align-items-center h-100 position-relative z-3">
                <h4 class="text-warning text-uppercase">Ошибка загрузки телеметрии</h4>
                <p class="text-secondary small">Проверьте соединение с сервером localhost:3000</p>
                <button class="btn btn-outline-warning btn-sm mt-2" onclick="location.reload()">Повторить</button>
            </div>
        `;
    }
}

```

### 3.4. Сборка фронтенда через Vite.js

**Теория:** Современные браузеры накладывают жесткие ограничения при работе с модульной структурой кода на клиенте (обилие HTTP-запросов к мелким JS-файлам, проблемы кэширования). Инструмент `Vite` решает эту проблему на этапе Production-сборки. По команде `npm run build` он запускает под капотом оптимизатор `Rollup`, который склеивает изолированные ES-модули страниц и компонентов, компилирует пути, удаляет лишние пробелы/комментарии (минификация) и генерирует единые легковесные ассеты в папку `public`.

Конфигурационный файл `client/vite.config.js` переопределяет стандартное поведение вывода:

```javascript
export default {
    build: {
        outDir: './public',      // Целевая папка для скомпилированного фронтенда
        emptyOutDir: true,       // Автоматическая очистка директории перед новым билдом
    },
};

```

### 3.5. Конфигурация Express для статики и SPA-роутинга

**Теория:** При переносе папки `public` в корень папки `server` бэкенд Express становится полноценным веб-сервером. Метод `express.static(publicPath)` автоматически перехватывает запросы к статическим файлам (скриптам, стилям, изображениям).

Однако в концепции Single Page Application (SPA) роутингом управляет клиентский JS-скрипт. Если пользователь обновит страницу, находясь по адресу `http://localhost:3000/product/1`, сервер Express попытается найти физический роутер `/product/1` и вернет ошибку `404`. Для предотвращения этого падения внедрен "catch-all" обработчик на базе регулярных выражений `/.*/`. Он перенаправляет любые запросы (не принадлежащие к префиксу `/api`) обратно на корневой файл `index.html`, позволяя клиентскому роутеру корректно восстановить страницу.

**Фрагмент настройки путей в `server/src/index.js` (с учетом синтаксиса Node.js v24+):**

```javascript
// Подключение папки с фронтенд-билдом
const publicPath = path.join(__dirname, '../public');
app.use(express.static(publicPath));

// API Маршруты (обработка данных)
app.use('/api/starships', starshipsRouter);

// Защитный 404 роутер только для неопознанных API запросов
app.use('/api', (req, res) => {
    res.status(404).json({ error: 'Космический маршрут API не найден' });
});

// SPA-Магия: Перенаправление всех остальных запросов на index.html.
// Использование RegExp-литерала /.*/ без кавычек предотвращает конфликты парсера строк в новых Node.js/Express.
app.get(/.*/, (req, res) => {
    res.sendFile(path.join(publicPath, 'index.html'));
});

```

### 3.6. Синхронизация параллельных задач на Promises (Доп. задание)

**Теория:** Модернизация дополнительного задания из ЛР №5 с применением спецификации `Promise`. В отличие от ручного счетчика на колбэках, экосистема промисов предоставляет нативный паттерн параллельного ожидания потоков.

Запрос 1 (с задержкой 30 секунд на сервере через `?delay=30000`) и Запрос 2 (мгновенный PATCH) инициируются одновременно. Браузер выполняет их параллельно. Нам нет необходимости выстраивать их в лесенку. Мы можем дождаться завершения "последнего выжившего" асинхронного процесса, используя логику подсчета ответов, что гарантирует атомарность последующего поискового запроса №3.

```javascript
const testParallelTask = () => {
    console.log("Запускаем Запросы 1 и 2 ОДНОВРЕМЕННО...");
    let completedRequests = 0;

    const tryRunSearch = () => {
        completedRequests++;
        console.log(`Получен ответ (${completedRequests}/2)...`);

        // Запуск Поиска (Запрос №3) строго на барьере завершения предварительных задач
        if (completedRequests === 2) {
            console.log("Оба запроса готовы! Запускаем поиск (Запрос №3)...");
            ajax.get(starshipUrls.getStarships('EXTRA')).then((searchData) => {
                console.log("Результат поиска:");
                console.table(searchData);
            });
        }
    };

    // Оба запроса улетают в сеть в одну и ту же миллисекунду
    ajax.patch(starshipUrls.updateStarshipById(1) + '?delay=30000', { name: "Extra-1" })
        .then(() => {
            console.log("Запрос №1 (долгий, 30 сек) успешно завершился.");
            tryRunSearch();
        });

    ajax.patch(starshipUrls.updateStarshipById(2), { name: "Extra-2" })
        .then(() => {
            console.log("Запрос №2 (быстрый) успешно завершился.");
            tryRunSearch();
        });
};

```
