## 3. Теория и реализация (Примеры из кода)

### 3.1. Изоляция сетевого слоя: Модуль Ajax

**Теория:** Написание низкоуровневого кода `XMLHttpRequest` внутри визуальных компонентов нарушает принцип единственной ответственности (Single Responsibility Principle). Для изоляции HTTP-протокола был создан вспомогательный сервисный класс `Ajax`, который инкапсулирует создание экземпляра `XMLHttpRequest`, настройку заголовков (например, `Content-Type: application/json` для отправки структур), отслеживание стадий выполнения запроса через `onreadystatechange` и безопасный парсинг `responseText` в JSON-формат.

**Пример реализации метода `PATCH` в `modules/ajax.js`:**

```javascript
class Ajax {
    patch(url, data, callback) {
        const xhr = new XMLHttpRequest();
        xhr.open('PATCH', url);
        // Явное указание формата передаваемых данных серверу
        xhr.setRequestHeader('Content-Type', 'application/json');
        xhr.send(JSON.stringify(data));

        xhr.onreadystatechange = () => {
            // readyState === 4 означает, что операция полностью завершена
            if (xhr.readyState === 4) {
                this._handleResponse(xhr, callback);
            }
        };
    }

    _handleResponse(xhr, callback) {
        try {
            const data = xhr.responseText ? JSON.parse(xhr.responseText) : null;
            callback(data, xhr.status); // Передача десериализованных данных в UI слой
        } catch (e) {
            console.error('Ошибка парсинга JSON:', e);
            callback(null, xhr.status);
        }
    }
}
export const ajax = new Ajax();

```

### 3.2. Маршрутизация URL-адресов эндпоинтов

Для централизованного управления путями к бэкенду разработан класс `StarshipUrls`. Это гарантирует, что при смене порта или домена сервера изменения вносятся локально в одном файле.

```javascript
class StarshipUrls {
    constructor() {
        this.baseUrl = 'http://localhost:3000/api';
    }

    getStarships(name = '') {
        return name ? `${this.baseUrl}/starships?name=${encodeURIComponent(name)}` : `${this.baseUrl}/starships`;
    }

    getStarshipById(id) {
        return `${this.baseUrl}/starships/${id}`;
    }

    updateStarshipById(id) {
        return `${this.baseUrl}/starships/${id}`;
    }
}
export const starshipUrls = new StarshipUrls();

```

### 3.3. Компонентная архитектура и жизненный цикл страниц

**Теория:** Интерфейс построен по принципу Single Page Application (SPA). Главный контейнер (`#root`) динамически очищается и наполняется разметкой классов страниц. Компоненты страниц (`MainPage`, `ProductPage`) выступают в роли координаторов: они инициируют сетевые запросы, управляют состояниями загрузки (рендеринг спиннеров) и передают чистые данные дочерним компонентам UI (`ProductCardComponent`, `EditFormComponent`) для генерации HTML-кода.

**Схема жизненного цикла страницы редактирования (`ProductPage`):**

1. Вызов метода `render()` $\rightarrow$ Очистка контейнера, отрисовка базового скелета страницы.
2. Автоматический вызов `getData()` $\rightarrow$ Отрисовка спиннера Bootstrap, запуск асинхронного `ajax.get()`.
3. Получение ответа от API $\rightarrow$ Очистка спиннера, делегирование рендеринга информации о корабле и активация формы через `editForm.render(item)`.

```javascript
// Фрагмент ProductPage из файла pages/product/index.js
getData() {
    this.pageRoot.innerHTML = `<div class="spinner-border text-light" role="status"></div>`;

    ajax.get(starshipUrls.getStarshipById(this.id), (data, status) => {
        if (status === 200 && data) {
            this.pageRoot.innerHTML = ''; // Скрытие спиннера

            // Рендерим блок статической информации
            const product = new ProductComponent(this.pageRoot);
            product.render(mappedData);

            // Рендерим форму и подкладываем текущие значения из БД
            const editForm = new EditFormComponent(this.pageRoot);
            editForm.render(data, this.saveChanges.bind(this));
        }
    });
}

```

### 3.4. Политика CORS и её преодоление в Dev-режиме

**Теория:** Браузерная политика ограничения одинакового источника (Same-Origin Policy) запрещает скрипту, загруженному с одного домена (`http://127.0.0.1:5500` из Live Server), запрашивать ресурсы с другого домена/порта (`http://localhost:3000`), если сервер явно не прислал разрешающие заголовки.

Для симуляции успешной интеграции без изменения конфигурации продакшн-сервера использовалось расширение **CORS Unblock**. Оно перехватывает HTTP-запросы на лету и внедряет в ответы заголовки `Access-Control-Allow-Origin: *`. Для обеспечения работы сложных запросов (таких как `PATCH` или `POST` с типом `application/json`), требующих предварительной верификации через метод `OPTIONS` (**preflight request**), в расширении были принудительно активированы опции перехвата `Access-Control-Request-Headers` и подмены статус-кодов `4xx` на `200 OK`.

### 3.5. Реализация параллельной асинхронности через счетчик (Доп. задание)

**Теория:** Главная сложность параллельного выполнения асинхронных функций на колбэках заключается в отсутствии встроенного механизма отслеживания общего состояния системы. Если запустить Запрос №1 и Запрос №2 одновременно, они выполнятся в фоновом потоке браузера. Запрос №2 вернется мгновенно, а Запрос №1 — через 30 секунд. Если вызвать Запрос №3 (Поиск) линейно, он улетит сразу (на 0.2 секунде) и вернет неактуальные данные, так как первый корабль еще не обновился.

Для решения этой проблемы был разработан паттерн **"Барьер" (или ручной счетчик завершения)**. Мы инициируем Запрос №1 и Запрос №2 одновременно на 0-й секунде. Каждый запрос по завершении вызывает общую изолированную функцию `tryRunSearch()`, увеличивая внутренний счетчик `completedRequests` на 1. Условие `completedRequests === 2` сработает строго тогда, когда придет ответ от самого медленного запроса (в данном случае, от Запроса №1 на 30-й секунде), что гарантирует актуальность базы данных перед выполнением итогового поиска.

```javascript
const testParallelTask = () => {
    console.log("Запускаем Запросы 1 и 2 ОДНОВРЕМЕННО...");
    let completedRequests = 0;

    // Функция-барьер для синхронизации потоков ответов
    const tryRunSearch = () => {
        completedRequests++;
        console.log(`Получен ответ (${completedRequests}/2)...`);

        // Только когда выполнены ОБА обновления, открывается шлюз для поиска
        if (completedRequests === 2) {
            console.log("Оба запроса готовы! Запускаем поиск (Запрос №3)...");
            ajax.get(starshipUrls.getStarships('EXTRA'), (searchData) => {
                console.log("Результат поиска (Запрос №3 выполнен):");
                console.table(searchData); // Логирование финальной таблицы изменений
            });
        }
    };

    // Запрос №1: Долгий (Активирует задержку setTimeout 30с на бэкенде)
    ajax.patch(starshipUrls.updateStarshipById(1) + '?delay=30000', { name: "Extra-1" }, (res) => {
        console.log("Запрос №1 (долгий, 30 сек) успешно завершился.");
        tryRunSearch();
    });

    // Запрос №2: Быстрый (Выполняется и возвращается мгновенно)
    ajax.patch(starshipUrls.updateStarshipById(2), { name: "Extra-2" }, (res) => {
        console.log("Запрос №2 (быстрый) успешно завершился.");
        tryRunSearch();
    });
};

```
