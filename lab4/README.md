## 1. Цель работы

Освоение серверной разработки на платформе `Node.js`. Создание собственного REST API веб-сервиса с использованием микрофреймворка `Express.js`. Изучение слоистой архитектуры приложения (Layered Architecture), принципов маршрутизации (Routing) и работы с протоколом HTTP. Тестирование эндпоинтов с помощью инструмента Postman.

## 2. Задание

### 2.1. Основное задание

Реализация на `Node.js` собственного веб-сервиса для предоставления `REST API`. Данные должны храниться атомарно в `json` файле без вложенных массивов. Приложение должно быть строго адаптировано под индивидуальную предметную область (Космические корабли Starship).

Требуется реализовать 5 базовых методов обработки HTTP-запросов:

* **GET** `/api/starships` — получение списка всех кораблей (с поддержкой фильтрации по имени).
* **GET** `/api/starships/:id` — получение данных конкретного корабля по его ID.
* **POST** `/api/starships` — создание новой записи.
* **PATCH** `/api/starships/:id` — частичное обновление данных корабля.
* **DELETE** `/api/starships/:id` — удаление корабля из базы.

### 2.2. Дополнительное задание (Реализация PUT)

Усложненное задание:

1. Изучить и описать теоретическую разницу между HTTP-методами `PUT` и `PATCH`.
2. Реализовать дополнительный 6-й эндпоинт — метод `PUT` для полного обновления (перезаписи) ресурса по ID, добавив соответствующую бизнес-логику в сервис и строгую валидацию в контроллер.

## 3. Теория и реализация (Примеры из кода)

### 3.1. Слоистая архитектура и Middleware

**Теория:** Express.js является нестрогим фреймворком и не навязывает конкретную структуру каталогов. Для обеспечения масштабируемости проекта была применена слоистая архитектура (Layered Architecture), разделяющая код на Роутеры (`routes`), Контроллеры (`controllers`) и Сервисы (`services`). Основой Express является концепция `Middleware` — промежуточных функций, через которые последовательно проходит объект запроса (`req`) перед тем, как попасть в контроллер.

**Пример из кода (Точка входа `index.js` и настройка Middleware):**

```javascript
const express = require('express');
const app = express();

// 1. Встроенный middleware для парсинга тела запроса в формате JSON
app.use(express.json());

// 2. Кастомный логирующий middleware (Access Log)
app.use((req, res, next) => {
    console.log(`[${new Date().toISOString()}] ${req.method} ${req.url}`);
    next(); // Передача управления следующему слою
});

// 3. Подключение роутера предметной области
app.use('/api/starships', starshipsRouter);

// 4. Глобальный обработчик ошибок (Error Handler)
app.use((err, req, res, next) => {
    console.error(err);
    res.status(500).json({ error: 'Внутренняя ошибка сервера телеметрии' });
});
```

### 3.2. Разница между PUT и PATCH (Доп. задание)

**Теория:** Оба метода используются для изменения существующих данных на сервере, но их семантика принципиально отличается:

* **PATCH (Частичное обновление):** Клиент отправляет только те поля, которые нужно изменить. Остальные данные ресурса остаются нетронутыми (происходит слияние старого и нового объектов).
* **PUT (Полная замена):** Идемпотентный метод, который требует отправки **полного** представления ресурса. Если клиент отправляет PUT-запрос и не указывает какое-либо поле, сервер обязан удалить это поле (заменить ресурс целиком, сохранив только его идентификатор).

В связи с этим, валидация в контроллере для этих методов существенно отличается.

### 3.3. Маршрутизация и Контроллеры

**Теория:** Роутер перехватывает HTTP-запрос по определенному URL и передает его в Контроллер. Задача Контроллера — извлечь данные из параметров (`req.params`), строки запроса (`req.query`) или тела (`req.body`), провести поверхностную валидацию, вызвать слой бизнес-логики и сформировать HTTP-ответ (с правильным статус-кодом).

**Пример из кода (Валидация PUT-запроса в `starshipController.js`):**

```javascript
const replaceStarship = (req, res) => {
    const id = parseInt(req.params.id);
    const { name, generation, status, commissionDate } = req.body;

    // Жесткая валидация: для PUT-запроса клиент ОБЯЗАН передать все поля
    if (!name || !generation || !status || !commissionDate) {
        return res.status(400).json({
            error: 'Для PUT запроса необходимы ВСЕ поля: name, generation, status, commissionDate'
        });
    }

    const replacedStarship = starshipService.replace(id, { name, generation, status, commissionDate });

    if (!replacedStarship) {
        return res.status(404).json({ error: 'Космический корабль не найден' });
    }
    res.json(replacedStarship);
};
```

### 3.4. Бизнес-логика и хранение данных (Сервисы)

**Теория:** Слой сервисов изолирует работу с данными (базой данных или файловой системой). Контроллеры ничего не знают о том, как физически хранятся данные. В рамках лабораторной работы в качестве хранилища используется атомарный JSON-файл `starships.json`.

**Пример из кода (Логика слияния `PATCH` против перезаписи `PUT` в `starshipService.js`):**

```javascript
// Реализация метода PATCH (Частичное обновление)
const update = (id, shipData) => {
    const starships = fileService.readData(dataFilePath);
    const index = starships.findIndex(s => s.id === id);
    if (index === -1) return null;

    // Слияние (Merge): старые данные перекрываются переданными новыми
    starships[index] = { ...starships[index], ...shipData };
    fileService.writeData(dataFilePath, starships);
    return starships[index];
};

// Реализация метода PUT (Полная замена)
const replace = (id, shipData) => {
    const starships = fileService.readData(dataFilePath);
    const index = starships.findIndex(s => s.id === id);
    if (index === -1) return null;

    // Замена: формируем абсолютно новый объект, сохраняя только оригинальный ID
    starships[index] = { id, ...shipData };
    fileService.writeData(dataFilePath, starships);
    return starships[index];
};
```
